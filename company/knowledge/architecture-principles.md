# 架构原则

## SOLID 原则

- 单一职责（SRP）：每个模块/服务只负责一个业务能力
- 开闭原则（OCP）：通过扩展而非修改来增加功能，善用接口和策略模式
- 里氏替换（LSP）：子类必须能完全替代父类使用
- 接口隔离（ISP）：接口要小而专，客户端不应依赖它不需要的方法
- 依赖倒置（DIP）：高层模块不依赖低层模块，都依赖抽象

## DRY 与适度抽象

- 不要重复自己（DRY），但避免过早抽象
- 规则：重复出现 3 次以上再考虑抽象
- 共享代码通过内部包/库管理，而非复制粘贴
- 抽象层次要一致，不要在同一函数中混合高低层操作

## 微服务 vs 单体

默认策略：单体优先，按需拆分

- 新项目从模块化单体（Modular Monolith）开始
- 满足以下条件时考虑拆分微服务：
  - 团队规模超过 8 人且需要独立部署节奏
  - 模块间有明确的限界上下文边界
  - 存在显著不同的扩缩容需求
- 拆分前必须先在单体中实现清晰的模块边界

## API 设计

- 对外 API 遵循 RESTful 规范，资源命名使用复数名词
- 内部服务间通信优先使用 gRPC
- 所有 API 必须版本化（URL 路径或 Header）
- 响应格式统一，包含 code、message、data 三个字段
- 分页、过滤、排序使用统一的查询参数规范

## 数据架构

- 每个服务拥有自己的数据库，禁止跨服务直接访问数据库
- 服务间数据同步通过事件驱动（Event-Driven）实现
- 关键数据变更必须发布领域事件
- 读写分离：复杂查询场景使用 CQRS 模式

## 可观测性

- 所有服务必须暴露健康检查端点
- 统一使用结构化日志（JSON 格式）
- 关键链路必须有分布式追踪（Trace ID 贯穿全链路）
- 核心业务指标必须有监控面板和告警规则
