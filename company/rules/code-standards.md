# 编码规范

## 基本原则

- 代码是写给人看的，其次才是给机器执行的
- 所有代码必须通过 CI 检查后才能合并
- 禁止提交含有 TODO/FIXME 的代码到主分支，除非附带对应的 Issue 链接

## 命名规范

- 变量/函数：camelCase（JS/TS）、snake_case（Python）
- 类名/组件名：PascalCase
- 常量：UPPER_SNAKE_CASE
- 文件名：kebab-case（前端）、snake_case（后端）
- 命名必须自解释，禁止使用 a、b、tmp 等无意义名称
- 布尔变量以 is/has/should/can 开头

## 函数与模块

- 单个函数不超过 40 行，超过必须拆分
- 函数参数不超过 4 个，超过请使用对象/结构体封装
- 每个模块/文件只做一件事（单一职责）
- 公共函数必须有文档注释，说明参数、返回值和异常情况

## 测试要求

- 新功能必须附带单元测试，覆盖率不低于 80%
- Bug 修复必须附带回归测试
- 关键业务逻辑要求集成测试覆盖
- 测试命名格式：`should_预期行为_when_条件`

## Code Review 规则

- 每个 PR 至少需要 1 位 Reviewer 批准
- 核心模块变更需要 2 位 Reviewer
- Review 重点关注：逻辑正确性、边界处理、安全隐患、性能影响
- PR 描述必须包含：变更目的、影响范围、测试方式
- 单个 PR 不超过 400 行变更，超过需拆分

## 代码格式

- 使用项目统一的 Linter 和 Formatter 配置
- 提交前必须运行 `lint` 和 `format`
- 禁止在 PR 中关闭 lint 规则，除非有充分理由并经 Review 同意
