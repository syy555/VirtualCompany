# 安全策略

## 核心原则

- 安全是每个人的责任，不仅仅是安全团队的事
- 默认最小权限原则
- 纵深防御，不依赖单一安全措施

## 密钥与敏感信息

- 严禁在代码中硬编码任何密钥、密码、Token、API Key
- 所有敏感配置通过环境变量或密钥管理服务（如 Vault）注入
- `.env` 文件必须加入 `.gitignore`，仓库中只保留 `.env.example`
- CI/CD 中的密钥使用平台提供的 Secret 管理功能
- 发现泄露的密钥必须立即轮换，不能仅靠删除 commit

## 依赖安全

- 每周自动运行依赖漏洞扫描（Dependabot / Snyk）
- 高危漏洞必须在 48 小时内修复或制定缓解方案
- 禁止引入已知存在严重漏洞的依赖
- 新增依赖需评估：维护活跃度、许可证兼容性、安全历史
- 锁定依赖版本，禁止使用 `*` 或 `latest`

## OWASP Top 10 防护要求

- 注入攻击：所有用户输入必须参数化查询，禁止拼接 SQL
- 认证失效：使用成熟的认证框架，强制 MFA 用于管理后台
- 敏感数据泄露：传输层强制 TLS，存储层加密敏感字段
- XXE：禁用外部实体解析
- 访问控制：API 层面强制鉴权，前端隐藏不等于安全
- 安全配置：生产环境关闭 debug 模式和详细错误信息
- XSS：输出编码，使用 CSP 头，前端框架自动转义
- 反序列化：不信任外部序列化数据
- 已知漏洞组件：见依赖安全章节
- 日志监控：关键操作必须记录审计日志，配置异常告警

## 安全事件响应

- 发现安全问题立即上报至 #security 频道
- 严重漏洞启动应急响应流程，2 小时内组建响应小组
- 事后必须进行复盘并输出改进措施
